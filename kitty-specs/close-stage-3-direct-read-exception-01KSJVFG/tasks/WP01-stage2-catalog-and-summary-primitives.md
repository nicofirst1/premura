# WP01 - Stage 2 Catalog and Summary Primitives

## Branch Strategy

- Planning/base branch: `master`
- Final merge target: `master`
- Implementation command after task finalization: `spec-kitty agent action implement WP01 --agent <name>`
- Execution worktree assignment is computed later from `lanes.json`; do not invent a worktree path manually.

## Objective

Create the Stage 2 foundation that closes the direct-read exception cleanly: typed engine result envelopes and engine helper functions for metric catalog entries and metric validity summaries.

## Context

- The spec requires `list_metrics` and `metric_summary` to stop reading fact tables directly from Stage 3.
- Current engine result types in `src/premura/engine/_results.py` cover question signals, not catalog/summary service outputs.
- Current public engine surface in `src/premura/engine/__init__.py` is lazy-loaded and tests explicitly lock that behavior.
- This WP must not change any MCP entrypoint registration yet; it should only create the engine contract that later WPs consume.

## Subtask Guidance

### T001 - Add typed Stage 2 result envelopes for metric catalog and metric summary

**Purpose**

Extend `src/premura/engine/_results.py` with explicit typed envelopes for:

- metric catalog entry
- metric validity summary

These types should reuse existing freshness vocabulary rather than inventing a second status model.

**Implementation notes**

- Reuse `FreshnessState` for `current`, `stale`, `unavailable`.
- Prefer typed dataclasses or the prevailing pattern already used in `_results.py`.
- Include fields agreed in planning:
  - catalog: `metric_id`, `validity_status`, `validity_window`, `missing_data_policy`, `latest_observation_at`, `latest_value`, `unit`, `message`
  - summary: above plus `sample_size`, `imputed_proportion`, `gap_count`, `window_days`
- Numeric fields must support honest absence. If the metric is unavailable or unknown, the envelope should be able to serialize without fabricated numbers.

**Watch-outs**

- Do not put MCP-specific prose or field names in the engine types.
- Do not include all-time extrema fields.

### T002 - Implement engine helper for validity-gated metric catalog entries

**Purpose**

Create a Stage 2 helper that can enumerate metrics and compute a validity-gated catalog entry for each one.

**Implementation notes**

- Place the helper behind the public engine surface in `src/premura/engine/__init__.py`, or a tightly scoped supporting module that `__init__.py` owns.
- Use the existing query/freshness primitives already present in the engine.
- The catalog should expose declared metadata (`validity_window`, `missing_data_policy`, canonical unit) plus a computed validity status and latest usable observation.
- No raw row-count field should survive this layer.

**Expected behavior**

- fresh metric -> `current`
- present but stale metric -> `stale`
- empty metric / unknown metric -> `unavailable`

**Watch-outs**

- Unknown metric IDs and known-but-empty metrics should both be handled honestly, but you may need a distinct message even if they share `unavailable`.
- Keep the helper deterministic and non-diagnostic.

### T003 - Implement engine helper for per-metric validity summaries over a fixed 30-day window

**Purpose**

Provide a Stage 2 summary helper that returns the latest usable value plus explicit coverage and imputation metadata over a recent 30-day window.

**Implementation notes**

- Use a fixed `window_days = 30`.
- Compute:
  - latest usable value
  - observation timestamp
  - validity status
  - sample size inside the recent window
  - imputed proportion
  - gap count
- Respect missing-data policy semantics. If a metric forbids imputation, `imputed_proportion` should remain `0.0`.

**Watch-outs**

- Do not regress the current freshness model.
- Do not surface all-time min/max/avg.

### T004 - Add engine tests for current/stale/unavailable catalog and summary semantics

**Purpose**

Prove the new helpers behave correctly before MCP consumes them.

**Files**

- `tests/test_engine.py`

**Test cases to add**

- fresh metric returns `current`
- stale metric returns `stale`
- empty metric returns `unavailable`
- unknown metric returns `unavailable` with no fabricated numeric fields
- summary returns 30-day `sample_size`, `imputed_proportion`, `gap_count`
- no-imputation policy yields `imputed_proportion == 0.0`

**Testing discipline**

- Follow public-interface testing only.
- Avoid peering into internal implementation state beyond what the public helper returns.

### T005 - Extend lazy-load contract coverage for the new engine helpers

**Purpose**

Protect the existing lazy-load guarantee while exposing the new helpers.

**Watch-outs**
