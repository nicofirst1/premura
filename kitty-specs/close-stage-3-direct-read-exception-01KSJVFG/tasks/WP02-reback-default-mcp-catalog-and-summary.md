# WP02 - Re-back Default MCP Catalog and Summary

## Branch Strategy

- Planning/base branch: `master`
- Final merge target: `master`
- Implementation command after task finalization: `spec-kitty agent action implement WP02 --agent <name>`
- This WP depends on WP01. Execution worktree assignment is computed later from `lanes.json`.

## Objective

Replace the last default-surface direct reads in `src/premura/mcp/server.py` by consuming the new Stage 2 catalog and summary helpers, while keeping all six existing signal-backed tools stable.

## Context

- Raw tool behavior currently lives in `src/premura/mcp/server.py`.
- The direct-read breach for `list_metrics` and `metric_summary` exists here, not in the engine.
- `_serialize_signal_result()` already handles signal envelopes; this WP needs parallel serialization for the new catalog/summary result types without regressing signal paths.
- `query_warehouse` still exists during this WP; its surface split happens in WP03.

## Subtask Guidance

### T006 - Rewire `list_metrics` to consume Stage 2 catalog helpers

**Purpose**

Stop building the default metric catalog from direct SQL counts and instead serialize Stage 2 catalog entries.

**Implementation notes**

- Replace fact-table count / metadata passthrough logic with calls to the new engine helper from WP01.
- The MCP payload should preserve machine readability for downstream callers.
- Keep the tool name stable unless the spec or contracts demand otherwise.

**Watch-outs**

- No raw row-count field should remain.
- Keep messages non-diagnostic and factual.

### T007 - Rewire `metric_summary` to consume Stage 2 summary helpers

**Purpose**

Make `metric_summary` return the new validity/imputation envelope instead of all-time aggregates.

**Implementation notes**

- Call the Stage 2 summary helper and serialize its output directly.
- Ensure fields like `sample_size`, `imputed_proportion`, and `gap_count` are explicit top-level result fields.
- Remove or stop emitting all-time min/max/avg style fields.

**Watch-outs**

- Unknown metrics and known-but-empty metrics must not fabricate `latest_value`.

### T008 - Serialize machine-branchable validity/imputation fields with honest absence semantics

**Purpose**

Make the MCP payload useful to downstream callers without parsing prose.

**Implementation notes**

- Ensure the serialized shape carries the structured fields agreed in the contract files.
- Distinguish `stale` from `unavailable` explicitly.
- Preserve clear explanatory `message` text, but do not rely on the message as the only signal.

**Watch-outs**

- Avoid leaking internal engine type details into the user-facing message.
- Keep consistency with the shipped four-state signal-backed tool payload style where appropriate.

### T009 - Add MCP payload tests for fresh/stale/empty/unknown catalog and summary responses

**Purpose**

Lock the new payload shapes through the public MCP tool surface.

**Files**

- `tests/test_mcp_signal_tools.py`

**Test cases to add**

- fresh catalog entry
- stale catalog entry
- empty catalog entry
- unknown summary request
- summary with explicit `sample_size`, `imputed_proportion`, and `gap_count`
- absence of all-time extrema fields

### T010 - Re-run and tighten signal-tool regression coverage

**Purpose**

Make sure moving catalog/summary through Stage 2 does not damage the already-shipped signal-backed tools.

**Implementation notes**

- Extend or adjust focused regression cases in `tests/test_mcp_signal_tools.py`.
- Confirm the six signal-backed tools still serialize through the same statuses and result shapes they did before.

## Test Strategy


Then combine with WP01 engine coverage when both are present.

## Definition of Done

- `list_metrics` no longer exposes raw fact-table count semantics on the default surface.
- `metric_summary` no longer exposes all-time extrema on the default surface.
- Both tools return explicit validity/imputation fields.
- Existing signal-backed tools retain their behavior.

## Risks

- Payload churn breaking downstream code that expected the old summary shape.
- Over-coupling new catalog/summary serialization to internal helper implementation details.
