# WP03 - Separate Operator Entrypoint and Raw SQL

## Branch Strategy

- Planning/base branch: `master`
- Final merge target: `master`
- Implementation command after task finalization: `spec-kitty agent action implement WP03 --agent <name>`
- This WP depends on WP01. It can run in parallel with WP02 after WP01 is available.

## Objective

Split the MCP registration surface so the default entrypoint is fully agent-safe while a separate operator entrypoint explicitly exposes the raw SQL escape hatch.

## Context

- `src/premura/mcp/entrypoint.py` currently registers all 9 tools in one `build_server()` function.
- The product decision is not to hide raw SQL behind config, but to expose a separate operator entrypoint.
- The operator entrypoint is lower-guarantee and may be used by an agent only after explicit user approval. That policy should be reflected in the exposed surface language, even though conversational approval enforcement lives outside the server.

## Subtask Guidance

### T011 - Refactor MCP registration into shared core plus explicit default/operator builders

**Purpose**

Separate common registration from surface-specific registration so the product has two clean entrypoints over one shared core.

**Implementation notes**

- Keep a shared registration core in `src/premura/mcp/entrypoint.py`.
- Build one default agent-facing surface and one operator surface.
- Avoid duplicating large registration blocks.

**Watch-outs**

- The default entrypoint must not accidentally retain `query_warehouse`.

### T012 - Add a separate operator entrypoint that registers `query_warehouse`

**Purpose**

Expose a distinct operator-only MCP command that adds `query_warehouse` to the otherwise safe tool surface.

**Implementation notes**

- The operator entrypoint should share the same process model and server core.
- It should differ by the extra raw SQL tool registration, not by a second warehouse implementation.

### T013 - Wire packaging / command entrypoints

**Purpose**

Make the new operator surface invokable explicitly.

**Files**

- `pyproject.toml`

**Implementation notes**

- Add a second console-script entrypoint for operator mode.
- Keep the existing `premura-mcp` command stable.

### T014 - Add tool-list tests for default vs operator surfaces

**Purpose**

Lock the registration policy with public-surface tests.

**Files**

- `tests/test_mcp_server.py`

**Test cases to add**

- default entrypoint omits `query_warehouse`
- operator entrypoint includes `query_warehouse`
- exact difference between the two surfaces is one tool

### T015 - Encode lower-guarantee operator mode language

**Purpose**

Make the operator surface clearly identifiable as lower-guarantee expert mode.

**Implementation notes**

- Use entrypoint naming and/or tool descriptions to make the boundary visible.
- Do not imply Stage 2 validity guarantees for the raw SQL path.
- Reflect the explicit-approval requirement as policy guidance, even if enforcement remains outside the server.

- Too much divergence between the two surfaces in this mission.
