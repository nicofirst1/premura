# Clone-first onboarding, with a uvx no-clone trial, and a durable XDG data dir

> **Status:** Accepted — 2026-07-15

The onboarding arc needs a human to install Premura and reach its first-run onboarding in any MCP-capable harness (Claude Code, OpenCode, Codex). Premura is operated _and improved_ from a local clone, so onboarding is clone-first: the operator keeps a checkout they can extend (add a parser, fix a mapping) and run their own edits live. Four decisions are locked. (1) **The primary path is a copy-paste agent prompt that clones the repo and sets it up** - the human pastes a short instruction into their coding agent, which clones `git+https://github.com/nicofirst1/premura`, runs `uv run premura bootstrap` (setup-only: prepare and verify the checkout and bundled skills), then `uv run premura install-client <harness>` to register the default surface. A uvx-from-git form (`uvx --from git+…/premura premura install-client <harness>`, launching `premura-mcp`) is retained as a no-clone _trial_ path for someone who only wants to try Premura, not as the headline: it runs the published code from uv's disposable cache, so clone edits never reach it. The client set is a `CLIENTS` registry, not a per-app ladder (DOCTRINE rule 2). (2) **The warehouse defaults to a durable XDG path, never the checkout** - `data_dir` → `$XDG_DATA_HOME/premura` (config/keys → `$XDG_CONFIG_HOME/premura`). A repo-relative default (the prior `REPO_ROOT/data`) would put special-category health data under the checkout; keeping it XDG-durable decouples where code lives from where data lives, and in particular keeps the warehouse off uv's garbage-collectable cache when the no-clone trial form is used. `PREMURA_DATA_DIR` still overrides, and an existing repo-local warehouse is preserved by pointing that variable at it or moving the directory once. (3) **`install-client` registers the clone-local server from a clone, and the uvx published form otherwise** - run from a Premura checkout (detected by a `pyproject.toml` whose `[project] name` is `premura`), it writes `uv run --directory <clone> premura-mcp` so the operator's own edits go live; run from a cold directory it writes the portable `uvx --from …/premura premura-mcp`. The `--directory` pin makes the clone form cwd-independent, which matters for the global codex config. This supersedes the earlier choice to always emit the uvx form and have a developer re-register by hand: clone-first makes running the clone's code the common case, not an expert afterthought. Rejected alternatives stand: an npm/npx wrapper shelling into Python (wrong ecosystem, two runtimes for one server), and gating install on a PyPI publish (uvx-from-git removes that gate; a later PyPI release only shortens the trial line to `uvx premura-mcp`). (4) **Only the validity-gated default surface is ever auto-registered** - `premura-mcp-operator --ack` (the raw-SQL escape hatch) stays a deliberate, user-approved manual step, kept out of every installer path, one-liner, and quick-start so it can never be enabled by reflex; `install-client`'s launch target is fixed to `premura-mcp` in both the clone and uvx forms, with no code path that emits `--ack`. Concept lineage: this realizes onboarding-arc gap #1; stage boundary is Stage 4 (UI/runtime).
