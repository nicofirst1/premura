# One-command install via uvx-from-git, and a durable XDG data dir

> **Status:** Accepted — 2026-07-15

The onboarding arc needs a human to install Premura and reach `/premura` in any MCP-capable harness (Claude Code, OpenCode, Codex) with **one command and no clone**. Four decisions were locked. (1) **The portable launch command is `uvx --from git+https://github.com/nicofirst1/premura premura-mcp`** — uvx fetches, builds an isolated cached env, and runs the default MCP surface straight from the public repo, so no git clone and no PyPI publish are required (uv is the only prerequisite; this is the Serena pattern for local-first Python MCP servers). This command is the harness-agnostic atom: each client differs only in the config syntax that wraps it, so `premura install-client <harness>` writes it into the right file (`.mcp.json` / `opencode.json` / `~/.codex/config.toml`) as an idempotent merge, and the installer itself can run clone-free via `uvx --from … premura install-client <harness>`. The client set is a `CLIENTS` registry, not a per-app ladder (DOCTRINE rule 2). (2) **The warehouse defaults to a durable XDG path, never the checkout** — `data_dir` → `$XDG_DATA_HOME/premura` (config/keys → `$XDG_CONFIG_HOME/premura`), because a uvx-launched server's code lives in uv's _disposable_ cache; a repo-relative default (the prior `REPO_ROOT/data`) would put special-category health data somewhere uv can garbage-collect. The ephemeral-env and persistent-data concerns are thereby decoupled; `PREMURA_DATA_DIR` still overrides, and an existing repo-local warehouse is preserved by pointing that variable at it or moving the directory once. (3) **`install-client` always emits the published uvx form; a developer registers `uv run premura-mcp` manually** — the clone variant uses local uncommitted edits and is documented in [`AGENT_CLIENTS.md`](../../using/AGENT_CLIENTS.md), rather than having the installer auto-detect clone-vs-cold and branch (rejected as magic). Also rejected: an npm/npx wrapper shelling into Python (wrong ecosystem, two runtimes for one server) and gating any one-command install on a PyPI publish (uvx-from-git removes that gate; a later PyPI release only shortens the line to `uvx premura-mcp`). (4) **Only the validity-gated default surface is ever auto-registered** — `premura-mcp-operator --ack` (the raw-SQL escape hatch) stays a deliberate, user-approved manual step, kept out of every one-liner, installer path, and quick-start so it can never be enabled by reflex; `install-client`'s launch args are fixed to `premura-mcp` with no code path that emits `--ack`. Concept lineage: this realizes onboarding-arc gap #1; stage boundary is Stage 4 (UI/runtime).
