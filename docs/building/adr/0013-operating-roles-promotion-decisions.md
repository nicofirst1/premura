# Operating-roles promotion: the five locked decisions

> **Status:** Accepted — 2026-06-12

The 2026-06-12 maintainer design-interview promoted the operating-agent-roles draft to a specification ([`src/premura/ui/OPERATING_ROLES.md`](../../../src/premura/ui/OPERATING_ROLES.md)) by locking five decisions. (1) **The orchestrator is a hybrid**: the operating agent stays the intelligence, while a thin deterministic layer owns the two things that must not depend on agent goodwill — the answer-audit gate and the handoff trace; roles are declarations in a bounded registry, never a hardcoded router. (2) **The audit gate blocks**: a presentation tool refuses to bless any health-interpreting draft that has not passed `answer_audit`, so anything presented as Premura-verified was audited and anything unaudited is visibly not verified — the gate cannot stop an agent from typing, so the guarantee is scoped to Premura's surface and bound by the runtime contract. (3) **The v1 verdict is deterministic**: code-verified honesty checks gate the answer (trace exists, measured disclosure attached, refusals not hidden); the AI rubric (research-trace-audit skill) runs on top as advisory only, promotable later. Citation-to-fetch binding is named slice-2 work because PubMed tools do not record into the research trace yet. (4) **Handoff records live in the session log database** (the sole-writer store of ADR 0011), keeping orchestrator events out of the research trace so multiplicity counts ("N hypotheses examined") stay uncontaminated by design. (5) **The first build slice** is the role-declaration registry, the handoff trace, and the blocking `answer_audit` gate; the improvement queue, share-packet flow, and lifestyle-context capture are later slices. Concept lineage: decision note 0010; runtime build-and-use boundary unchanged.
