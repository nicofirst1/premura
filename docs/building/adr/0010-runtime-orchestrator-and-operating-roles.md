# Runtime orchestrator and operating roles

> **Status:** Accepted — 2026-06-02

Premura's runtime multi-agent surface will distinguish the **orchestrator** from **operating roles**: the orchestrator routes a human's health-data goal, sequences role handoffs, and records a compact trace, while declaration-driven roles such as `ingest`, `analysis`, `human_facing`, `answer_audit`, and `improvement_scan` own bounded responsibilities and governance surfaces. Cross-role handoffs go through the orchestrator, health-interpretive answers are audited before final narration, and product gaps found during operation become private improvement candidates before any user-approved issue or dev-time workflow. This keeps runtime operation separate from bootstrap/setup, preserves traceability without hidden role memory, and avoids turning runtime roles into persona names or an unbounded router switch. The orchestrator/operating-role separation is narrow: an "operating role" is a job the orchestrator dispatches through Premura's MCP tools, and parser-building stays outside that set as ordinary file-editing. That separation does **not** gate local use — at runtime an agent may build a parser and use it immediately for the operator's own data with no reviewer, as part of using an installed Premura; only contributing that parser back as a public PR enters the gated dev-time/review path.
